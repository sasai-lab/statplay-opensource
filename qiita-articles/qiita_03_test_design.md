## 「動いた、よし」で進めていた時期

[前回](https://qiita.com/sasailab)、ガンマ関数の実装の話で「検証したから正しい」と書きました。じゃあその「検証」って何をやっているのか。今回はその話です。

最初は、テストなんて書いていませんでした。

AIにコードを書いてもらって、ブラウザで開いて、グラフが動いて、「動いた。よし」で次に進む。正直に言うと、しばらくはそれで進んでいました。

転機は正規分布の累積分布関数を実装したときです。グラフはそれらしく描けている。でもふと思ったんです。

**「この計算、本当に合ってるのか？」**

統計を勉強していると `normCDF(1.96) = 0.975` という値は嫌というほど出てきます。正規分布表の1.96のところ。これが間違っていたら、このツールで学んだ人が間違った理解をすることになる。

コンソールで叩いてみたら `0.9750021...` と出ました。合ってそう。でも「合ってそう」で済ませていいのか。もっともらしいけど微妙にズレている、みたいなことがあったら？　気づけるのか？

ここから、テストを書き始めました。

---

## 「正解」をどうやって決めるか

テストを書こうとして、最初にぶつかったのがこの問題でした。

AIに「この関数の正しい値を教えて」と聞けば、値は出てきます。でもちょっと考えると、それはおかしい。**AIが書いたコードを、AIが出した正解で検証したら、同じ間違いをしていても気づけない。** カンニングの答え合わせをカンニング元でやっているようなものです。

じゃあどうするか。AIとは別の情報源から「正解」を持ってこなきゃいけない。

ここでAIに「独立した参照元が欲しいんだけど」と相談したら、「数値計算の古典的な参照テーブルとしてAbramowitz & Stegunという本があります」と教えてくれました。1964年に出た本で、数値計算の分野では権威のある参照値集だそうです。正直この本の存在を初めて知りましたが、そこに載っている `erf(1) ≈ 0.8427` という値をテストの正解として使いました。

あとは、もっとシンプルなものもあります。

- **統計検定で暗記する値**: `normCDF(1.96) ≈ 0.975` は教科書に載っている定数。これは自分が勉強していたから「正しいはず」と言える
- **数学的に絶対に成り立つ関係**: 前回書いた `Γ(5) = 24`（= 4!）や `Γ(0.5) = √π`。こういう恒等式は参照元すら要らない。数学的に正しいことが保証されている
- **関数同士の整合性**: 累積分布関数を微分したら確率密度関数になるはず。実際に数値微分して比較する。これが一致しなかったら、どちらかが壊れている

こうやって「正解の根拠」を探していく過程が、思いのほか面白かったです。テストを書くために統計の定義を改めて調べ直すことになって、結果的に自分の理解も深まりました。

## 実際のテストコード

言葉で書いても伝わりにくいので、実際のコードを見せます。

```javascript
// 統計検定で暗記する値で検証する
const z975 = normCDF(1.96);
const z990 = normCDF(2.326);
const z995 = normCDF(2.576);
ok('normCDF reference: Φ(1.96)≈0.975; Φ(2.326)≈0.990; Φ(2.576)≈0.995',
   Math.abs(z975 - 0.975) < 5e-4 &&
   Math.abs(z990 - 0.990) < 5e-4 &&
   Math.abs(z995 - 0.995) < 5e-4);
```

`normCDF(1.96)` の正解が `0.975` であることは、正規分布表を見れば誰でも確認できます。AIが出した値ではない。ここが大事なところです。

Abramowitz & Stegunの参照値を使ったテストはこうです。

```javascript
// 1964年刊の数値計算参照テーブルから持ってきた値
ok('erf(1) ≈ 0.84270 (Abramowitz & Stegun reference)',
   Math.abs(erf(1) - 0.8427007929) < 5e-4);
```

数学的に絶対に成り立つ関係を使ったテストもあります。

```javascript
// Γ(5) = 4! = 24、Γ(0.5) = √π — これは数学的に保証されている
ok('gamma: Γ(1)=1, Γ(5)=24, Γ(0.5)≈√π',
   Math.abs(gamma(1) - 1) < 1e-10 &&
   Math.abs(gamma(5) - 24) < 1e-8 &&
   Math.abs(gamma(0.5) - Math.sqrt(Math.PI)) < 1e-8);
```

前回の記事で書いた `Γ(5) = 4! = 24` がここで使われています。あのとき「なんで $\Gamma(5) = 5!$ じゃなくて $(5-1)!$ なのか」と首を傾げた知識が、テストの正解を自分の言葉で書ける根拠になっている。理解してから書くテストは、コピペのテストとは違います。

## `5e-4` と `1e-10` の違い

テストコードを見て気づいた人もいるかもしれません。許容誤差がテストごとに違います。`5e-4`（0.0005）のところもあれば、`1e-10`（0.0000000001）のところもある。

最初は全部 `1e-6` くらいにしておけばいいだろうと思っていました。でもAIに「なんで統一しないの」と聞いたら、理由がありました。

`Γ(1) = 1` や `Γ(5) = 24` は整数の階乗です。数学的に厳密な正解がある。だから `1e-10` まで詰められる。一方、`normCDF(1.96) ≈ 0.975` は正規分布表の値で、そもそも表自体が小数点以下3〜4桁。正解の精度以上にテストを厳しくしても意味がない。だから `5e-4` で十分。

**正解の精度に合わせてテストの精度を決める。** 言われてみれば当然なんですが、自分では気づけませんでした。「テストの許容誤差にも根拠がいる」ということを、テストを書きながら学びました。

---

## 「信頼する」と「鵜呑みにする」は違う

AIはコードを書いてくれます。テストコードも書いてくれます。でも「何をテストすべきか」「正解をどこから持ってくるか」は、自分で考えなきゃいけませんでした。

`normCDF(1.96) = 0.975` を正解にしようと思えたのは、統計検定の勉強をしていたから。Abramowitz & Stegunという参照元があることは、AIに教えてもらった。全部ひとりでは無理だったし、全部AIだけでも無理だった。

これで統計関数の正確性は検証できるようになりました。でもStatPlayはWebサイトです。計算が正しくても、UIが壊れていたら意味がない。次回は、サイト全体のQAの話です。
