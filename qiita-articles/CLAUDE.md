# Qiita連載「AI駆動開発で統計ツールを作る」制作ガイド

## 概要

StatPlayの開発知見をQiitaに連載する。
Qiitaへの投稿は手動。Claude Codeの作業範囲はマークダウンファイルの作成まで。
`statplay-opensource` リポジトリへの記事ファイルのプッシュは禁止。

## 目的

1. **StatPlayの権威性の確立** — 技術的解像度の高さで「AIに投げっぱなしではない」ことを証明
2. **AI駆動開発の実践者としてのポジション確立** — 非エンジニアがAIと協業して本番品質を作れることを示す

## トンマナ

### ポジション

- 「自分で考え出したのではなく、AIに手伝ってもらいながら理解した」を常に前面に出す
- 「自分はすごくない。やり方さえ知れば誰でもできる」を崩さない
- 「AIが書いたから正しいはず」ではなく「AIと一緒に検証したから正しい」
- 自分を過大評価しない。「気づいたら理解できていた」という謙虚な語り口

### 文体

- 一人称は「私」
- 「最初に言っておくと、私はエンジニアではありません。」という書き出しパターンを基本とする
- 「AIに聞いたら教えてくれた」「一緒に考えながら理解した」を自然に入れる
- 断定的に書く。「〜だと思います」より「〜です」
- 体験談として書く。「〜でした」という過去形で進める
- コードブロックには必ずコメントを入れる

### 等身大のトーンを維持するためのルール

- 機能紹介は「こういうことができます」ではなく「自分はこれで腹落ちしました」のように自分の体験として書く
- 教訓めいた語り口にしない。「〜しなければいけません」より「〜でした」「〜だった」と体験ベースで書く
- 苦労や失敗を隠さない。「正直かなり苦労しました」「高校数学から学び直す羽目になった」のように弱さを見せる
- 「凄まじい」「圧倒的」など読者に複雑・困難な印象を与える表現は避ける。「気づいたら〜になっていました」くらいに留める
- 「誰でもできる」という連載の約束と矛盾する表現を入れない
- 権威的・総括的なまとめ文（「〜を実感しました」「〜が重要です」）で段落を締めすぎない。体験の描写で終わるほうが等身大になる

### 章立てのルール

- **毎回違う構成にする。** 「はじめに → 問題 → 説明 → 検証 → 学び」のような固定パターンを繰り返さない。人間が書いた連載は回ごとに構成が違う。テンプレートに従っている時点でAIっぽくなる
- 見出しは記事の内容から自然に決める。前回と同じ見出し（「はじめに」「結局、何が学びだったか」等）を使い回さない
- 導入に「## はじめに」が必要とは限らない。noteブロックの直後に本文が始まってもいい
- 締めに「## 結局〜」「## まとめ」を毎回置かない。記事ごとに自然な終わり方を探す

### 分量のルール

- 1記事の目安は**本文2000〜3000字**（コードブロック・数式除く）。これを超えそうなら品質を落とさず記事を分ける
- 「長いけど削れない」と感じたら、テーマを絞って2本に分けることを提案する。無理に1本にまとめて薄くするより、濃い記事を2本にするほうがいい
- 分割時は前の記事の末尾で次回予告し、次の記事の冒頭で前回を受ける

### やってはいけないこと

- AIが書いたコードをそのまま貼り付けて終わりにしない
- 検証なしに「これが正しい実装です」と断言しない
- エンジニア向けの専門用語を説明なしに使わない
- StatPlayのURLを貼るだけで終わりにしない（自然につながる文脈を作る）

## 記事の構成

### 固定要素（すべての記事に含める）

```markdown
---
title: [記事タイトル]
tags:
  - AI駆動開発
  - StatPlay
  - [内容に応じたタグを3個まで追加]
private: false
updated_at: ''
id: null
organization_url_name: null
slide: false
ignorePublish: false
---

:::note info
この記事は、統計学習ツール [StatPlay](https://statplay.sasailab.com) を開発する中で
学んだことをまとめたものです。AIと協業して開発していますが、数学的な実装の正確性は
自分なりに理解・検証しています。完璧かと言われると、まだ学びの途中です。
:::
```

- 末尾に関連リンク（StatPlay / GitHub）と著者署名を置く
- 連載の前後関係がある場合は前回・次回へのリンクを含める

### 含めるべき要素（見出し・順序は毎回変える）

- **導入**: なぜこのテーマに向き合うことになったか
- **問題**: StatPlayを作る中でぶつかった具体的な問題
- **技術**: 概念の説明 → 実際のコード → コードの解説（コメント必須）
- **検証**: テストコードと参照値。「AIが書いたから正しい」ではなく「検証したから正しい」
- **締め**: 技術的な学びとStatPlayとしての学びの接続

これらを「## はじめに → ## 問題 → … → ## まとめ」の固定構成にしない。記事の内容に合わせて自然な章立てを毎回考える。

```

例:
qiita_01_lgamma.md
qiita_02_test_design.md
qiita_03_zero_dependency.md
```

### フォーマット

- Qiitaのfrontmatter（冒頭の `---` ブロック）を必ず含める
- コードブロックには言語指定（```javascript）を必ず付ける
- 数式はKaTeX記法（`$$`）を使う
